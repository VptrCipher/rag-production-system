# Vector Databases: A Practical Guide

## What is a Vector Database?

A vector database is a specialised database designed to store, index, and search high-dimensional numerical vectors efficiently. These vectors are typically dense embedding representations of text, images, audio, or other unstructured data generated by machine learning models.

Unlike traditional relational databases that excel at exact lookups, vector databases perform **Approximate Nearest Neighbour (ANN)** search — finding the K vectors most similar to a query vector in a fraction of the time it would take to compare all vectors exhaustively.

## Why Vector Databases?

Traditional databases fail at semantic search because:
- SQL `LIKE` queries match on surface text, not meaning
- Exact key-value lookups require knowing the exact query
- Millions of vectors cannot be compared exhaustively in real time

Vector databases solve this with purpose-built ANN indices (HNSW, IVF, etc.) and similarity metrics (cosine, dot product, Euclidean distance).

## Key Concepts

### Embeddings
Numerical representations of data in a high-dimensional space. Text embeddings from models like `text-embedding-3-small` (OpenAI) or `all-MiniLM-L6-v2` (Hugging Face) encode semantic meaning so that similar texts cluster together in vector space.

### Similarity Metrics
- **Cosine Similarity**: Angle between vectors. Best for normalised embeddings. Range: [0, 1].
- **Dot Product**: Magnitude × cosine similarity. Used when vectors are not normalised.
- **Euclidean Distance (L2)**: Direct distance in space. Less preferred for text.

### ANN Algorithms
- **HNSW (Hierarchical Navigable Small World)**: Graph-based, fast queries, good recall. Default in Qdrant, Weaviate.
- **IVF (Inverted File Index)**: Cluster-based, good for very large datasets.
- **ANNOY**: Tree-based, read-optimised, no updates. Used by Spotify.
- **ScaNN**: Google's product quantisation approach, very fast for large scale.

## Major Vector Databases

### Qdrant
**Type**: Open-source, purpose-built vector DB (Rust)  
**Strengths**: High performance, filtering support, local disk mode, easy Docker setup  
**Use case**: Production RAG systems, hybrid search  
**Storage**: In-memory or disk-based (`QdrantClient(path="...")`)  
**Key feature**: Rich filtering on metadata payload before/during vector search

### Pinecone
**Type**: Managed cloud vector DB  
**Strengths**: Zero infrastructure management, serverless tier, very easy API  
**Use case**: Startups, rapid prototyping, production at scale  
**Weakness**: Vendor lock-in, cost at scale, no free self-hosting  
**Pricing**: Free tier (1 project), scales with storage and queries

### Weaviate
**Type**: Open-source, schema-based vector DB (Go)  
**Strengths**: GraphQL API, multi-modal support, built-in modules (BM25, transformers)  
**Use case**: Complex schema-driven RAG, knowledge graphs  
**Key feature**: Hybrid search with built-in BM25 + vector fusion

### Chroma
**Type**: Open-source, embedded vector DB (Python-first)  
**Strengths**: Extremely easy to set up, great for development and prototyping  
**Use case**: Local development, Jupyter notebooks, small-scale RAG  
**Weakness**: Not production-grade at scale, limited filtering

### FAISS (Facebook AI Similarity Search)
**Type**: Library (not a database), open-source (C++/Python)  
**Strengths**: Fastest pure ANN search, GPU support, battle-tested  
**Use case**: Research, batch similarity search, building custom vector stores  
**Weakness**: No persistence, no filtering, no server — must build around it

### Milvus
**Type**: Open-source, distributed vector DB  
**Strengths**: Scales to billions of vectors horizontally, Kubernetes-native  
**Use case**: Enterprise scale, multi-tenancy, very large corpora  
**Weakness**: Complex to operate, heavy infrastructure requirements

### pgvector
**Type**: PostgreSQL extension for vector operations  
**Strengths**: No new infrastructure — existing Postgres deployment, SQL interface  
**Use case**: Teams already on Postgres, simple RAG with <1M vectors  
**Weakness**: Not as fast as dedicated vector DBs at scale

## Qdrant Deep Dive (Used In This Project)

### Collections
A collection is a named set of vectors with the same dimensionality. Equivalent to a database table.

```python
from qdrant_client import QdrantClient
from qdrant_client.http.models import Distance, VectorParams

client = QdrantClient(path="local_qdrant_storage")
client.create_collection(
    collection_name="rag_documents",
    vectors_config=VectorParams(size=384, distance=Distance.COSINE),
)
```

### Payload Filtering
Qdrant supports filtering on metadata before or during ANN search (pre-filtering):

```python
from qdrant_client.http.models import Filter, FieldCondition, MatchValue

results = client.search(
    collection_name="rag_documents",
    query_vector=embedding,
    query_filter=Filter(
        must=[FieldCondition(key="file_type", match=MatchValue(value="pdf"))]
    ),
    limit=10,
)
```

### Disk-Based Storage
For development without Docker:
```python
client = QdrantClient(path="./local_qdrant_storage")
```

For production with Docker:
```python
client = QdrantClient(host="localhost", port=6333)
```

## Vector Index Tuning

### HNSW Parameters
- `m`: Number of bi-directional links per node (default: 16). Higher = better recall, more memory.
- `ef_construct`: Size of dynamic candidate list during construction (default: 100). Higher = better quality index, slower build.
- `ef`: Search time parameter. Higher = better recall, slower search.

### Quantisation
For large-scale deployment, quantise vectors to reduce memory:
- **Scalar quantisation**: 4× memory reduction with ~1% recall drop
- **Product quantisation**: 8–16× reduction with larger recall trade-off

## Best Practices

1. **Always normalise embeddings** before storing if using cosine similarity
2. **Use batch upserts** (256–512 points per batch) for efficient ingestion
3. **Set HNSW ef parameter** high (128–256) for better recall at query time
4. **Shard large collections** across nodes for horizontal scalability
5. **Use payload indexes** on frequently filtered metadata fields for fast filtering
6. **Monitor memory usage** — HNSW indices are memory-intensive
